# Seller Hub

Seller Hub — личный кабинет продавца с AI-ассистентом для улучшения объявлений.

## Описание

Приложение покрывает основной пользовательский поток продавца:

`/ads` → `/ads/:id` → `/ads/:id/edit` → `save` → обратно на деталку.

В проекте реализованы:

- список объявлений с поиском, сортировкой, фильтрами и пагинацией;
- страница просмотра объявления;
- страница редактирования объявления с валидацией, сохранением черновика и AI-помощью;
- интеграция с локальной LLM через Ollama:
  - улучшение / генерация описания;
  - рекомендация рыночной цены;
- тёмная тема;
- переключение layout списка: сетка / список.

## Структура

Проект разложен по слоям в духе FSD, но без избыточной декомпозиции страниц.

```text
frontend/src
├── app
│   ├── App.tsx
│   ├── main.tsx
│   ├── index.css
│   └── routes
├── entities
│   ├── ad
│   │   ├── api
│   │   ├── lib
│   │   ├── model
│   │   └── ui
│   └── ai-ollama
│       ├── api
│       └── model
├── pages
│   ├── ads-list-page
│   ├── ad-details-page
│   └── ad-edit-page
├── shared
│   └── api
└── widgets
    ├── ads-toolbar
    ├── ads-filters
    └── ads-list
```

Подход к страницам:

- `ads-list-page` разложена на `widgets`, потому что там много независимых блоков;
- `ad-details-page` и `ad-edit-page` оставлены более цельными, потому что их логика сильно локальна для конкретной страницы;
- для таких экранов использован pragmatic-подход: держать код ближе к `pages`, а не дробить всё насильно на `widgets/features`.

## Что сделано

### Основной функционал

**Список `/ads`**

- поиск по названию;
- сортировка;
- фильтрация по категориям;
- фильтр `Только требующие доработок`;

**Деталка `/ads/:id`**

- title, price, даты;
- описание и характеристики;
- блок `Требуются доработки`;
- переход назад к списку;
- переход на страницу редактирования.

**Редактирование `/ads/:id/edit`**

- форма по категориям `auto / real_estate / electronics`;
- required-валидация `title` и `price`;
- warning-подсветка пустых необязательных полей;
- `PUT /items/:id`;
- `Сохранить` / `Отменить`;
- toast на успешное и неуспешное сохранение;
- счётчик символов для описания;
- draft в `localStorage` с восстановлением после refresh.

## Принятые решения

### FAQ

**Почему выбраны Axios + TanStack Query?**

`Axios` здесь используется как удобный HTTP-клиент, а `TanStack Query` закрывает кэширование, loading/error states и повторные запросы. Для этого проекта такая связка оказалась практичнее, чем ручное управление сетевым state.

**Почему `GET /items` был расширен полем `id`?**

В исходном ответе списка не хватало идентификатора объявления для нормальной маршрутизации на `/ads/:id`. Поэтому backend был расширен так, чтобы список тоже возвращал `id`.

Без этого нельзя было бы корректно:

- перейти с карточки на деталку;
- строить стабильные `key`;
- связать список и редактирование.

**Какие сортировки реально поддерживаются?**

В проекте реализованы:

- `createdAt-desc`
- `createdAt-asc`
- `title-asc`
- `title-desc`

**Как работает draft в `localStorage`?**

- для edit-страницы используется ключ вида `ad-edit-draft-<id>`;
- draft сохраняется автоматически при изменении формы;
- при открытии `/ads/:id/edit` draft восстанавливается;
- после успешного сохранения draft удаляется;
- при `Отменить` пользователь возвращается на деталку без сохранения на сервер.

**Почему страницы не разложены по полной FSD-декомпозиции?**

Под дедлайн был выбран более прагматичный вариант:

- список `/ads` разложен на `widgets`, потому что там это реально даёт пользу;
- `ad-details-page` и `ad-edit-page` оставлены более цельными;
- локальная логика, не требующая переиспользования, не выносилась в отдельные `features/widgets` ради формальности.

## Как запускать

### Требования

- Node.js `20.19+` или `22.12+`
- npm
- Ollama

Важно: `frontend` использует Vite 8, поэтому для `npm run build` нужна версия Node не ниже `20.19`.

### 1. Запуск backend

```bash
cd server
npm i
npm start
```

Сервер по умолчанию стартует на `http://localhost:8080`.

### 2. Запуск frontend

```bash
cd frontend
npm i
npm run dev
```

Frontend по умолчанию стартует на `http://localhost:5173`.

### 3. Настройка Ollama

Нужно установить Ollama отдельно, вне репозитория.

После установки достаточно:

```bash
ollama pull qwen3:4b
ollama run qwen3:4b
```

Проект ожидает локальный Ollama API по адресу:

```text
http://127.0.0.1:11434/api
```

### 4. Проверка сценария

1. Открыть `/ads`
2. Перейти на деталку объявления
3. Перейти в edit
4. Изменить поля
5. Проверить draft после refresh
6. Запросить AI-описание / рекомендацию цены
7. Сохранить объявление
8. Убедиться, что произошёл редирект обратно на деталку
